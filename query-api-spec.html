Query filings, inspect cases, and retrieve filing documents.

The Filing / Case API is designed to let approved consumers search filings by status and metadata, search cases for activity, and navigate to related filing documents through predictable, versioned, ReST-style endpoints.

Base pattern /api/v1/...
Authentication API key over HTTPS
Primary format JSON request / response bodies

Principles

ReSTful

This API follows ReSTful architectural principles with noun-based URIs, standard HTTP methods, stateless requests, consistent JSON representations, and conventional HTTP status codes. Filtering, pagination, and predictable naming are applied where appropriate.

Responses include hypermedia links to related resources so consumers can move from search results to filing details and document retrieval without relying on hardcoded assumptions.

Authentication & authorization

Every request must include a valid API key. Keys should be unique per client, securely generated, rotatable, revocable, and never exposed in URLs, logs, or client-side code.

Authorization header header required

Preferred pattern: Authorization: ApiKey <key>

HTTPS transport security required

All requests must be transmitted over HTTPS to protect credentials in transit.

Usage controls policy platform-defined

Rate limiting, key scoping, rotation, and revocation should be enforced where applicable.

Example request 
curl https://integration.greenfiling.com/api/v1/filings/search \
  -X POST \
  -H "Authorization: ApiKey sk_test_123456" \
  -H "Content-Type: application/json" \
  -d '{
    "data": {
      "status": "REJECTED",
      "lastChangedTimestamp": "2026-01-01T00:00:01Z"
    },
    "metadata": {
      "pageNumber": 1,
      "pageSize": 25
    }
  }'

Versioning strategy

URI path versioning

Endpoints include an explicit version in the route, such as /api/v1/filings. Breaking changes require a new major version. Existing versions should remain available for a defined deprecation period so consumers can migrate cleanly.

Minor, non-breaking enhancements may be introduced within an existing version as long as the request and response contract remains backward compatible.

Use these routes to search filings, search cases, retrieve resource details, and navigate to documents.

POST /api/v1/filings/search

Search filings

Search for filings using a structured JSON body instead of a large query string. Results return filing objects with enough metadata to display status and navigate to related resources.

Wildcard search (*) supported for name-based inputs such as courtName.

Request body

data.status string optional

Filter by filing status, such as REJECTED or ACCEPTED.

data.lastChangedTimestamp datetime optional

Filter results to filings changed on or after a given timestamp in ISO 8601 UTC format.

data.caseNumber string optional

Filter by the case number associated with the filing.

data.envelopeNumber string optional

Filter by envelope number when a filing is already known externally.

data.courtName string optional

Filter by court name. Name-based matching should support wildcard search behavior.

metadata.pageNumber number required

The page number to return.

metadata.pageSize number required

The maximum number of results to return on a page.

Request example 
{
  "data": {
    "status": "REJECTED",
    "lastChangedTimestamp": "2026-01-01T00:00:01Z"
  },
  "metadata": {
    "pageNumber": 1,
    "pageSize": 25
  }
}

Filing search response

The response returns a data array of filing objects and a metadata object for pagination. Filing objects may include HATEOAS-style links to related resources such as filing documents.

Filing object fields

idstring

Unique filing identifier.

statusstring

Current filing status.

initiatedTimestampdatetime

Timestamp when the filing was initiated.

lastChangedTimestampdatetime

Timestamp of the most recent state change.

caseNumberstring

Case number tied to the filing.

caseNamestring

Friendly case name when available.

envelopeNumberstring

Envelope reference value.

courtNamestring

Name of the court.

caseTitlestring

Human-readable case title.

rejectionReasonstring

Present when the filing has been rejected and the reason is available.

linksarray

Related resources, typically with type, href, and httpMethod.

POST /api/v1/cases/search

Search cases

Search for cases using status, initiation time, last change time, or case name. Results can include a nested list of related filings so consumers can identify recent filing activity without separate lookups.

Wildcard search should be supported for caseName and other name-oriented lookups.

Request body

data.statusstringoptional

Case status filter, such as OPEN or CLOSED.

data.initiatedTimestampdatetimeoptional

Lower-bound filter for case initiation time in ISO 8601 UTC format.

data.lastChangedTimestampdatetimeoptional

Lower-bound filter for most recent case changes.

data.caseNamestringoptional

Case name filter with wildcard support.

metadata.pageNumbernumberrequired

The page number to return.

metadata.pageSizenumberrequired

The maximum number of case records to return.

Request example 
{
  "data": {
    "status": "OPEN",
    "initiatedTimestamp": "2026-01-01T00:00:01Z"
  },
  "metadata": {
    "pageNumber": 1,
    "pageSize": 25
  }
}

GET /api/v1/documents/{id}/content

Retrieve document

Returns the binary content of a document (e.g., a PDF file) resource by its unique identifier. This route is useful when a consumer moves from a filing search result into the canonical document record. This endpoint streams the document file directly. The response body contains raw binary data, not JSON.

Example request 
GET /api/v1/documents/doc_12345/content HTTP/1.1
Authorization: ApiKey sk_test_123456
Example response (Headers) 
HTTP/1.1 200 OK
Content-Type: application/pdf
Content-Length: 248193
Content-Disposition: attachment; filename="filing-instructions.pdf"

HTTP status codes

The API uses conventional HTTP status codes to communicate outcomes clearly and predictably.

Status Meaning Typical use
200 OK Request succeeded. Successful GET and POST search responses.
400 Bad Request The request payload or parameters are invalid. Malformed JSON, invalid paging values, or unsupported filters.
401 Unauthorized The request is missing valid authentication. Missing or invalid API key.
403 Forbidden The client is authenticated but not allowed to access the resource. Insufficient scope or blocked access.
404 Not Found The requested resource does not exist. Unknown case id or filing id.
500 Internal Server Error An unexpected server-side failure occurred. Unhandled exception or downstream system issue.
